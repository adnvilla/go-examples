# Wire

**Category:** design-patterns
**Difficulty:** Intermediate

## Objective

Show compile-time dependency injection with [google/wire](https://github.com/google/wire): instead of wiring constructors by hand in `main` (see [inject](../inject/)), a code generator reads a declarative injector function and produces the equivalent wiring as plain Go source.

## Concepts Covered

- `wire.Build(...)` — a declarative list of constructors the generator uses to figure out the dependency graph
- The `//go:build wireinject` build tag, which excludes the injector source (`wire.go`) from normal compilation — it exists only for the `wire` tool to read
- `wire_gen.go` — the generated file that's actually compiled and committed, containing ordinary hand-writable-looking Go code
- Contrast with [inject](../inject/): same dependency graph, generated instead of hand-assembled

## Prerequisites

- Go 1.25+
- No external services or environment variables required to **run** the example
- Regenerating `wire_gen.go` requires the `wire` CLI: `go install github.com/google/wire/cmd/wire@latest`

## Project Structure

```
wire/
├── go.mod
├── main.go
├── wire.go       (the injector — excluded from normal builds)
├── wire_gen.go   (generated — this is what actually compiles)
└── README.md
```

## How to Run

```bash
make run
# or
go run .
```

To regenerate `wire_gen.go` after changing `wire.go` (requires the `wire` CLI):
```bash
make generate
# or
wire gen ./...
```

## Expected Output

`Greeter.Grumpy` is set based on whether the current Unix timestamp is even or odd, so the output alternates unpredictably between runs:

```
hi there!
```
or, on roughly half of all runs:
```
Go away!
```

## Code Walkthrough

- `wire.go` declares `InitializeEvent(phrase string) Event` and calls `wire.Build(NewEvent, NewGreeter, NewMessage)` inside it — this tells `wire` "figure out how to construct an `Event`, given these three constructors, and generate the code for it." The function body (`return Event{}`) is never actually executed; it exists only to satisfy the compiler when analyzing this file, which is excluded from real builds by the `wireinject` build tag anyway.
- `wire_gen.go` is what `wire gen` produced: a straight-line `InitializeEvent` that calls `NewMessage`, then `NewGreeter`, then `NewEvent`, in dependency order — indistinguishable from what a human would write by hand for [inject](../inject/)'s style, except it's generated and guaranteed consistent with the declared graph.
- `main` calls `InitializeEvent("hi there!")` (resolved to the generated version, since `wire_gen.go` is the one actually compiled) and starts the event, which greets using whatever `Greeter.Grumpy` state `NewGreeter` decided based on the current time.
- `NewGreeter`'s grumpy/not-grumpy coin flip isn't about Wire at all — it's incidental to this example (adapted from the [Go Cloud](https://github.com/google/go-cloud) project) and just makes the output visibly vary between runs.

## Common Pitfalls

- **Hand-editing `wire_gen.go`.** It's marked `// Code generated by Wire. DO NOT EDIT.` for a reason — any manual change is silently discarded (and diverges from `wire.go`) the next time someone runs `wire gen`.
- **Forgetting the `wireinject` build tag on the injector file.** Without `//go:build wireinject`, `wire.go`'s `InitializeEvent` (whose body is just a stub) would conflict with `wire_gen.go`'s real implementation of the same function signature — the build tag is what lets both files coexist without a duplicate-definition error.
- **Adding a new constructor to `wire.Build(...)` without regenerating.** `wire.go` describes the *intended* graph; `wire_gen.go` is the *actual* compiled wiring — after editing `wire.go`, `wire_gen.go` is stale until `wire gen` runs again.
- **Expecting Wire to catch runtime errors.** Wire only wires *constructors* together at compile time — if a constructor is set up to return `(T, error)` and can fail, that's still an ordinary runtime error path, not something Wire itself detects at generation time.
